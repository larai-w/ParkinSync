作成日: 2026-07-07

## 0. 現状サマリー（As-Is）

ParkinSync は「パーキンソン病介護記録のデジタル化パイプライン」の MVP 段階にある。

```
紙の介護ログ（Word テンプレ / 手書き）
        │ 写真・スキャンを S3 にアップロード
        ▼
AWS Lambda（S3 トリガー）
        │ Amazon Textract で表構造 OCR
        │ Visual Crossing API で当日の気象データを付加（兵庫県の座標）
        ▼
Google Sheets に 13 列で自動追記（JST タイムスタンプ付き）
```

- 記録項目: 服薬（朝・昼・夕・就寝前）、排便/モビコール、転倒回数、緊急コール、体調スコア（1–5, OFF/ON）、服薬後の経時トラッキング、自由記述メモ
- 実装済み: Lambda 本体、単体テスト（moto 不使用の mock ベース）、Word ログテンプレ生成スクリプト、デプロイスクリプト
- 未実装: アップロード用フロントエンド、ユーザー管理、複数患者対応、OCR 精度検証フロー、データ可視化

**本質的な価値仮説:** 「介護者は紙に書くのが一番ラク。デジタル化の手間をゼロにすれば、医師・家族・ケアチームが使える構造化データが自然に貯まる」

## 1. 開発に使うスキルセット（Fable / Claude Code として）

### 1-1. 技術スキル領域

| 領域 | 具体的な技術 | 本プロジェクトでの用途 |
|---|---|---|
| サーバーレス設計 | AWS Lambda, S3, Secrets Manager, EventBridge, Step Functions | 現行パイプラインの堅牢化・リトライ設計 |
| OCR / ドキュメントAI | Amazon Textract (TABLES), Claude Vision API | 手書き文字の読取精度向上。Textract で拾えない手書き日本語は Claude Vision へのフォールバックが有力 |
| データ基盤 | Google Sheets API → 将来的に DynamoDB / RDS | MVP は Sheets 継続、患者数が増えたら DB 移行 |
| モバイル/フロント | React Native or Flutter、または LINE Mini App | 「写真を撮って送るだけ」の UX。介護者は高齢層が多いため LINE 連携が最短経路 |
| 医療系コンプライアンス | 3省2ガイドライン、個人情報保護法（要配慮個人情報）、薬機法の非該当設計 | 「診断・治療をしない記録支援ツール」に留めることで SaMD（医療機器プログラム）該当を回避 |
| データ可視化 | Looker Studio → 独自ダッシュボード | ON/OFF 変動と天気・服薬タイミングの相関可視化 |
| テスト/CI | pytest + moto, GitHub Actions, IaC (SAM/CDK) | 現在の unittest を拡張し、Textract レスポンスのゴールデンテスト追加 |

### 1-2. Claude Code のワークフロースキル（開発の進め方）

- **Plan mode / Plan エージェント**: フェーズごとの実装計画を先に固めてから着手（特に DB 移行やモバイル化のような不可逆な設計判断）
- **/code-review, /security-review**: 医療関連データを扱うため、毎 PR でセキュリティレビューを必須化（Secrets の扱い、S3 バケットポリシー、PII のログ出力禁止）
- **/verify, /run**: Lambda をローカル（SAM CLI）で実際に動かし、サンプル画像→Sheets 追記まで E2E で確認
- **Explore / general-purpose エージェント**: Textract のレスポンス仕様調査、規制要件のリサーチを並行実行
- **テスト駆動**: OCR は「入力のブレ」が本質的リスクなので、実際の手書きサンプル画像をテストフィクスチャとして蓄積し回帰テスト化

## 2. 開発戦略（ロードマップ）

### Phase 0: MVP 堅牢化（現在〜1.5ヶ月）— 「自分の家族で毎日使える」

- [ ] 日付パースの汎用化（現在 April/May のみ対応 → 通年 + 日本語日付）
- [ ] OCR 失敗時のリカバリ: 失敗画像を「要手動確認」フォルダに退避し通知（SNS/LINE Notify）
- [ ] Sheets 書き込みの重複防止（同じ画像の再処理でも二重登録しない冪等性）
- [ ] IaC 化（AWS SAM）と CI/CD（GitHub Actions）
- [ ] ハードコードされた座標・SpreadsheetID の設定化（環境変数 / DynamoDB 設定テーブル）
- 判断基準: **30日連続で手動介入ゼロで運用できたら Phase 1 へ**

### Phase 1: プロダクト化（2〜4ヶ月）— 「他の家族 10 世帯が使える」

- [ ] LINE Bot 化: 介護者が LINE でログ写真を送信 → 自動処理 → 結果を LINE に返信（「4/20 分を登録しました。転倒 1 回、注意」）
- [ ] マルチテナント化: 患者/世帯ごとのデータ分離（DynamoDB + 世帯ごとの Sheets or 統合 DB）
- [ ] OCR 精度向上: Textract 信頼度スコアが低いセルは Claude Vision で再読取 → それでも低ければ人間確認 UI
- [ ] 週次サマリーの自動生成: 「今週の ON 時間は先週比 +40 分。転倒は 2 回で雨の日に集中」（Claude API 活用）
- 判断基準: **知人・患者会経由で 10 世帯のパイロット、週次継続率 70% 以上**

### Phase 2: ケアチーム連携（5〜9ヶ月）— 「診察室で使われる」

- [ ] 医師向けレポート出力: 受診前に直近 4 週間の服薬・症状変動を A4 1 枚の PDF に（UPDRS 的な経時グラフ）
- [ ] ケアマネ・訪問看護師の閲覧権限（ロールベースアクセス）
- [ ] 服薬リマインダーと記録の突合（飲み忘れ検知）
- [ ] 神経内科クリニック 2〜3 院での臨床パイロット
- 判断基準: **医師から「これがあると診察が変わる」という定性フィードバック**

### Phase 3: スケール（10ヶ月〜）

- [ ] 他疾患への横展開（認知症、ALS など「介護記録が診療価値を持つ」疾患は構造が同じ）
- [ ] 匿名化データの研究利用（治験・レジストリ連携。ここが将来の主要収益源候補）
- [ ] 3省2ガイドライン準拠のインフラ監査、ISMS/プライバシーマーク検討

### 設計上の重要判断（先に決めておくこと）

1. **「紙ファースト」を捨てない。** アプリ直接入力への全面移行は失敗パターン。紙 → 写真の導線は主要 UX として維持し、直接入力は補助とする。
2. **医療機器該当性の回避。** 「症状の評価・診断・治療方針の提示」をしない。記録・可視化・共有に徹する。将来 SaMD 化するなら別プロダクトとして切り出す。
3. **Google Sheets は Phase 1 で「エクスポート先」に格下げ。** マスターデータは自前 DB に移す（要配慮個人情報を第三者 SaaS にマスター保管しない）。

## 3. ユーザーストーリー（詳細検討パート)

### ペルソナ

| ID | ペルソナ | 概要 |
|---|---|---|
| P1 | **陽子（62）** 主介護者 | 夫（68, PD 診断 6 年目, Yahr III）を在宅介護。スマホは LINE と写真のみ。紙のノートに毎日記録する習慣はある |
| P2 | **健太（35）** 遠方の息子 | 東京在住。母（P1）の負担と父の容態を把握したいが、電話では「大丈夫」しか返ってこない |
| P3 | **田中医師** 神経内科医 | 外来 1 人 8 分。診察室での「調子どうですか」「まあまあです」から服薬調整するしかないのが不満 |
| P4 | **佐藤さん** ケアマネジャー | 月 1 回のモニタリング訪問。転倒回数と排便状況が把握できるとサービス調整がしやすい |
| P5 | **研究コーディネーター** | 治験/観察研究で日々の症状日誌（患者日誌）の回収と転記に膨大な工数を割いている |

### Epic A: 記録のデジタル化（コア）

- **A-1** 陽子として、いつもの紙ログの写真を LINE で送るだけで記録が保存されてほしい。新しい操作を覚えたくないから。
  - 受入条件: LINE トーク画面で画像送信 → 60 秒以内に「登録完了/失敗」の返信。追加操作ゼロ。
- **A-2** 陽子として、読み取り結果が間違っていたら簡単に直したい。夫の服薬記録が間違ったまま医師に伝わるのが怖いから。
  - 受入条件: 返信メッセージに読取結果の要約を表示。「直す」タップで該当セルのみ修正できる。信頼度の低いセルは⚠️マーク付き。
- **A-3** システムとして、同じ写真が 2 回送られても重複登録しない。
  - 受入条件: 画像ハッシュ + 日付での冪等性チェック。重複時は「4/20 分は登録済みです」と返信。
- **A-4** 陽子として、月が変わったら新しい紙テンプレートが自動で手に入ってほしい。
  - 受入条件: 月末に翌月分の印刷用 PDF が LINE で届く（現行 `generate_word.py` の自動化）。

### Epic B: 家族間の見守り共有

- **B-1** 健太として、父の今週の状態サマリーを週 1 で受け取りたい。母に電話で聞き出す罪悪感なしに状況を知りたいから。
  - 受入条件: 毎週日曜に「今週: 転倒 1 回（雨の火曜）、ON 時間 平均 6.2h（先週 5.8h）、緊急コールなし」形式の LINE 通知。
- **B-2** 健太として、転倒や緊急コールがあった日は即日知りたい。
  - 受入条件: 該当項目が記録されたら 1 時間以内にアラート通知。閾値は世帯ごとに設定可能。
- **B-3** 陽子として、共有する範囲を自分で決めたい。排便の記録まで息子に見せたくはないから。
  - 受入条件: 項目単位の共有 ON/OFF 設定。デフォルトは「転倒・緊急コールのみ共有」。

### Epic C: 診療連携

- **C-1** 田中医師として、診察前に直近 4 週間の服薬遵守と ON/OFF 変動を 1 枚で見たい。8 分の診察で服薬調整の判断材料が欲しいから。
  - 受入条件: A4 1 枚 PDF。上段: 服薬時刻 × 体調スコアのヒートマップ。下段: 転倒・特記事項のタイムライン。診察予約日の前日に生成。
- **C-2** 陽子として、診察時に「レポートを見せる」だけで済ませたい。口頭で 1 ヶ月分を思い出して説明するのは無理だから。
  - 受入条件: LINE で「受診レポート」と送ると PDF が返ってくる。印刷 or 画面提示のどちらでも読めるレイアウト。
- **C-3** 田中医師として、天候と症状の相関が見たい（PD 患者は気圧・気温で変動する報告があるため）。
  - 受入条件: レポートに天気アイコン行を併記（既存の Visual Crossing 連携を活用）。※医学的判断はせず事実の並置に留める。

### Epic D: ケアチーム連携（Phase 2）

- **D-1** 佐藤さん（ケアマネ）として、担当利用者の月次の転倒・排便傾向を訪問前に確認したい。サービス調整の根拠になるから。
  - 受入条件: 閲覧専用アカウント。担当利用者のみ、家族が許可した項目のみ表示。
- **D-2** 陽子として、誰が自分たちの記録を見たか確認できるようにしたい。
  - 受入条件: アクセスログの本人開示（閲覧者・日時）。

### Epic E: 研究利用（Phase 3）

- **E-1** 研究コーディネーターとして、患者日誌を紙回収 → 手転記せず構造化データで受け取りたい。転記コストとエラーをなくしたいから。
  - 受入条件: 明示同意済み患者のデータを CDISC 系フォーマットでエクスポート。監査証跡付き。
- **E-2** 陽子として、研究提供への同意はいつでも撤回でき、撤回後のデータは使われないでほしい。
  - 受入条件: 同意管理画面 + 撤回時の提供停止処理。

### ストーリーの優先順位マップ

```
価値 高 ┃ A-1  A-2       C-1  C-2
        ┃ A-3       B-1
        ┃      B-2  B-3  C-3
        ┃ A-4            D-1  D-2
価値 低 ┃                     E-1  E-2
        ┗━━━━━━━━━━━━━━━━━━━━━━━
          実装コスト 低  →  実装コスト 高
```

MVP+（Phase 1）のスコープは **A-1〜A-4 + B-1** に絞る。C 系は「医師が受け取って嬉しい形式」のヒアリングが先。

## 4. マーケティング戦略

### 4-1. 市場規模の目安

- 国内パーキンソン病患者: 約 25〜30 万人（有病率は高齢化で増加傾向）
- うち在宅・家族介護でかつ記録習慣のある層を 10% と仮定 → **SAM ≈ 2.5〜3 万世帯**
- 横展開（認知症在宅介護 ≈ 数百万人規模）まで含めると TAM は 2 桁大きい

### 4-2. Go-to-Market（フェーズ連動）

| フェーズ | チャネル | 施策 |
|---|---|---|
| Phase 1 | 患者会（全国パーキンソン病友の会 都道府県支部） | パイロット 10 世帯の募集。会報への寄稿、体験会。**「開発者自身の介護経験」ストーリーが最強の信頼資産** |
| Phase 1–2 | 難病相談支援センター・保健所 | 難病医療費助成の申請窓口で介護者と接点がある。チラシ + QR |
| Phase 2 | 神経内科クリニック（B2B2C） | 医師向けレポート（C-1）を武器に「先生から患者さんに勧めてもらう」導線。導入 2–3 院の事例化 |
| Phase 2 | ケアマネ・訪問看護ステーション | 地域包括支援センターの勉強会、介護支援専門員協会 |
| Phase 3 | 製薬・CRO（研究利用） | PD 治療薬の市販後調査・治験の ePRO 代替として提案 |

### 4-3. 収益モデル

| モデル | 価格イメージ | 位置づけ |
|---|---|---|
| 家族向けフリーミアム | 無料: 記録 + 月次閲覧 / 有料 ¥500–980/月: 家族共有・週次サマリー・受診レポート | 継続利用の基盤。単体での黒字化は狙わない |
| クリニック向け | ¥5,000–20,000/月/施設（患者ダッシュボード） | Phase 2。医師の推奨が B2C 獲得も牽引 |
| 研究・データ提供 | 案件単位（数百万円/研究） | Phase 3。**本命の収益源**。同意・匿名化・監査証跡が参入障壁になる |

### 4-4. メッセージング

- 介護者向け: 「**書くのはいつものノートのまま。写真を撮るだけで、家族と先生に伝わる。**」
- 医師向け: 「『調子どうですか』の 8 分を、データのある 8 分に。」
- 訴求しないこと: 「AI が症状を診断」— 薬機法・信頼の両面で NG。あくまで記録支援。

### 4-5. KPI

- North Star: **週次アクティブ記録世帯数**（写真送信が週 4 日以上ある世帯）
- 補助: OCR 自動確定率（人手修正不要率 ≥ 90% 目標）、週次サマリー開封率、医師レポート発行数、パイロット継続率（3 ヶ月 ≥ 70%）

## 5. 開発コスト見積り

### 前提

- 開発体制: **開発者 1 名 + Claude Code（AI ペアプロ）**。従来の受託開発比で実装工数を 40–60% 圧縮できる想定
- 人件費レート: 個人開発の機会費用として ¥800,000/人月 で換算（外注する場合は 1.2〜1.5 倍を見込む）
- 為替: $1 = ¥150

### 5-1. 開発費（イニシャル）

| フェーズ | 期間 | 工数（AI 支援込み） | 人件費換算 | 主な内訳 |
|---|---|---|---|---|
| Phase 0: MVP 堅牢化 | 1.5 ヶ月 | 0.8 人月 | ¥64 万 | IaC 化、日付パース汎用化、冪等性、通知 |
| Phase 1: LINE Bot + マルチテナント | 3 ヶ月 | 2.5 人月 | ¥200 万 | LINE Messaging API、DynamoDB 設計、修正 UI、週次サマリー |
| Phase 2: 診療連携 | 4 ヶ月 | 3.5 人月 | ¥280 万 | PDF レポート、権限管理、クリニックダッシュボード、パイロット対応 |
| **小計（Phase 0–2）** | **約 9 ヶ月** | **6.8 人月** | **¥544 万** | |
| 外部コスト（Phase 0–2） | | | ¥60–120 万 | デザイン外注 ¥30–50 万、法務レビュー（利用規約・プライバシーポリシー・医療機器該当性確認）¥20–40 万、LINE 公式アカウント認証・雑費 |
| **イニシャル合計** | | | **約 ¥600–670 万** | 外注中心なら ¥900 万–1,200 万 |

（参考: 従来型の受託開発で同スコープを見積もると 12–15 人月 ≈ ¥1,200–1,800 万。AI 支援開発が前提のコスト構造）

### 5-2. ランニングコスト（月額）

| 項目 | 10 世帯（パイロット） | 300 世帯 | 3,000 世帯 |
|---|---|---|---|
| Textract（$15/1,000 頁, 世帯 30 頁/月） | $4.5 ≈ ¥700 | $135 ≈ ¥2.0 万 | $1,350 ≈ ¥20 万 |
| Lambda + S3 + DynamoDB | ¥500 未満 | ¥3,000 | ¥3 万 |
| Claude API（低信頼セル再読取 + 週次サマリー） | ¥1,000 | ¥2 万 | ¥15 万 |
| Visual Crossing（気象） | 無料枠 | ¥5,000 | ¥1.5 万 |
| LINE Messaging API | 無料枠 | ¥5,000 | ¥3 万 |
| 監視・その他（CloudWatch 等） | ¥500 | ¥3,000 | ¥2 万 |
| **月額合計** | **≈ ¥3,000** | **≈ ¥5.5 万** | **≈ ¥45 万** |
| 世帯あたり原価 | ¥300 | ¥180 | ¥150 |

→ 有料プラン ¥980/月なら粗利率 80% 超。**変動費構造上、価格設定は成立する。**

### 5-3. 損益分岐の目安

- 固定費（開発償却 ¥600 万 ÷ 24 ヶ月 + 運用保守 0.3 人月/月）≈ **¥49 万/月**
- 有料転換率 30% と仮定すると、¥980/月 プランで **約 1,700 登録世帯**（有料 510 世帯）で単月黒字
- ただし現実的にはクリニック契約（¥1–2 万/月 × 20 院 = ¥20–40 万/月）と研究案件が黒字化を前倒しする

### 5-4. コストに含めていないもの（要注意）

- 3省2ガイドライン完全準拠の監査・認証取得（Phase 3、数百万円規模）
- 治験 ePRO として使う場合のバリデーション（CSV 対応、別途見積り）
- マーケティング広告費（本戦略は患者会・医師推奨のオーガニック前提のため最小）

## 6. 主要リスクと対応
